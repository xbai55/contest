# A6 智慧校园 AI 原生赛题要求与 CampusPilot 对照

## 1. 赛题定位

赛题名称：

```text
智慧校园·AI原生
基于金蝶AI苍穹的新质校园智能体创新平台
```

CampusPilot 当前选择的方向：

```text
智能学业规划与预警 Agent
```

业务主线：

```text
学生画像
→ 课程成绩
→ 学习行为
→ 金蝶 Agent 风险分析
→ 风险预警单
→ 辅导员确认
→ 导师帮扶
→ 学生反馈
→ 复评结案
→ 驾驶舱治理分析
```

## 2. 比赛核心要求

| 要求 | 赛题含义 | CampusPilot 对应设计 |
| --- | --- | --- |
| 金蝶 AI 苍穹为核心底座 | 应体现低代码平台、Agent 平台和 OpenAPI 接入能力 | 金蝶侧承载 Agent、知识库、业务对象；本地前后端承载演示工作台和业务闭环 |
| 开发语言 Java | 赛题实现条件明确写明“开发语言：Java” | 本地业务后端改为 `CampusPilotServer.java`，Python 服务仅保留为开发期备份 |
| 真实校园痛点 | 不能只是泛泛的信息管理系统 | 聚焦学业风险识别、帮扶协同、复评结案 |
| 完整业务闭环 | 有角色、有流程、有数据沉淀 | 学生、辅导员、导师、学院管理者四类角色协同处理预警 |
| AI 原生能力 | 自然语言交互、RAG、工具调用、多步推理 | Agent 在金蝶平台低代码实现，本地系统提供 Agent API 调用入口和业务数据上下文 |
| 低代码应用能力 | 领域模型、表单、流程、报表、服务模型 | 学生画像、课程成绩、学习行为、风险预警单、处理日志等对象需要在金蝶平台建模 |
| 数据驱动决策 | 看板、报表、趋势、异常识别 | 驾驶舱展示风险分布、趋势、待办、帮扶成效 |
| 创新突破 | 多智能体、多模态、工作流等 | 可扩展为学业风险 Agent、成长规划 Agent、辅导员处置 Agent、治理分析 Agent |
| 非功能要求 | 流畅、美观、原创、模拟数据 | 本地页面已用模拟学生数据，禁止真实学生信息 |

## 5. 当前主要不足

| 优先级 | 不足 | 影响 | 建议 |
| --- | --- | --- | --- |
| P0 | 金蝶 Agent API 尚未真实接入 | AI 原生能力展示会弱 | 用金蝶平台 Agent 的真实 API 替换 Java 后端中的本地兜底回答 |
| P0 | 金蝶 OpenAPI 真实认证未完成 | 苍穹平台运用能力展示不足 | 配置第三方应用、AccessToken、API 授权、IP 白名单 |
| P0 | 金蝶低代码对象与本地字段映射需确认 | 数据模型说明书和演示容易断层 | 固化对象编码、字段标识、接口入参/出参 |
| P0 | PostgreSQL 尚未接入 | 赛题实现条件包含 PostgreSQL | 当前使用 Java 内存模拟数据，后续可用 JDBC 切换为 PostgreSQL |
| P1 | BPMN 正式流程未接入 | 业务闭环目前靠处理日志模拟 | 先用风险处理日志支撑演示，后续接入流程服务 |
| P1 | 数据样本仍偏少 | 驾驶舱分析说服力有限 | 扩展到更多学生、专业、周趋势和帮扶案例 |
| P1 | 测试报告尚未形成 | 初赛交付材料不完整 | 补功能用例、接口用例、角色用例截图 |
| P2 | 多智能体表达还不够明显 | 创新性分数提升有限 | 在 PPT 和 Agent 设计中拆分多个 Agent 职责 |

## 6. 后续开发顺序建议

1. 以 Java 后端作为正式本地服务：运行 `CampusPilotServer.java`，Python 服务只作为备份。
2. 接入真实金蝶 Agent API：保留本地 `agent/chat` 作为代理层，隐藏密钥和平台细节。
3. 完成第三方应用配置：`appId`、`appSecret`、代理用户、API 授权、IP 白名单。
4. 固化金蝶业务对象字段：学生画像、课程成绩、学习行为、风险预警单、风险处理日志。
5. 视时间接入 PostgreSQL：先保持模拟数据可演示，再用 JDBC 替换数据仓储层。
6. 做接口文档：本地前端 API、金蝶 OpenAPI、Agent API 三类接口分开写。
7. 扩充测试数据：至少覆盖高风险、关注、正常、改善、成长规划五类学生。
8. 完成测试报告和演示视频脚本。

## 7. 答辩表述建议

可以这样说明当前架构：

```text
CampusPilot 采用“金蝶平台智能体 + Java 本地业务工作台”的演示架构。
金蝶 AI 苍穹平台负责 Agent、知识库、工具编排和低代码业务对象；
Java 后端负责多角色工作台 API、风险预警闭环、驾驶舱分析和 Agent API 调用入口。
这样既能体现 AI 原生能力，也能保证比赛演示中的业务流程完整可运行。
```

一个简单例子：

```text
辅导员问：张明远为什么是高风险？
→ 本地系统把学生画像、课程成绩、学习行为作为上下文传给金蝶 Agent
→ 金蝶 Agent 返回风险解释和帮扶建议
→ 本地系统把建议落到风险预警单
→ 辅导员确认，导师填写措施，学生提交反馈，学院看驾驶舱成效
```
